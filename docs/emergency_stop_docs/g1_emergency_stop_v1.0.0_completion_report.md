## 2. 项目概述

本版本交付的是一个面向 Unitree G1 机器人的 ROS2 急停节点。程序以实体急停按钮为输入源，通过 GPIO 检测按钮按下/释放边沿，在检测到急停后优先调用高层 `Damp` 指令将机器人切换至阻尼模式，并辅以 TTS 语音、LED 灯光、日志输出和受控退出机制，形成一条从物理输入到高层保护动作的完整安全链路。

V1.0.0 的设计目标不是“程序尽快退出”，而是“在任何异常或退出场景下，尽量先把机器人置于更安全的状态，再结束当前进程”。

## 3. 版本目标与完成结论

### 3.1 版本目标

V1.0.0 目标如下：

1. 实现实体急停按钮的稳定检测与消抖。
2. 在急停触发后立即发起机器人阻尼切换。
3. 对阻尼切换结果进行 FSM 二次确认，避免仅凭 RPC 返回码误判。
4. 在启动、自检、急停触发、调试模式、保护失效、异常退出等场景下提供清晰提示。
5. 当检测链路故障或进程异常时，进入受控退出，而不是静默失效。
6. 将程序纳入 ROS2 工程构建与安装链路，并预留后台托管能力。

经源码检查、构建日志核对和安装产物确认，G1 急停程序 V1.0.0 已完成预期范围内的核心开发与工程集成，当前具备以下交付特征：

- 已具备 TTS、LED、日志三类运行提示能力。

## 4. 已完成功能清单

### 4.1 急停输入检测

- 基于 `libgpiod` 实现 GPIO 监听。
- 支持输入消抖参数 `debounce_ms`。
- 支持 `active_low` 电平极性适配。
- 检测器支持按下沿和释放沿回调分发。
- 启动阶段支持自检，避免在保护链路失效状态下继续运行。

### 4.2 启动自检与上电保护

- 节点启动后会执行 GPIO 自检。
- 若开机时急停已处于按下状态，程序会先补发一次急停动作，再等待按钮释放后自动重检。
- 若自检通过，会播报“急停自检通过”。
- 若自检失败或 GPIO 初始化异常，会触发“保护失效”流程并退出。

### 4.3 急停主动作

- 在检测到急停按下沿后，串行执行急停处理逻辑。
- 首次优先直接发送 `Damp` 指令，减少热路径额外等待。
- 支持失败后按参数进行重试。
- 每次 `Damp` 成功后，都通过 `GetFsmId` 校验当前是否真正进入阻尼模式。
- 仅在 FSM 确认成功后，才将急停动作判定为成功完成。

### 4.4 释放策略

- 急停释放后不会自动恢复运动。
- 若此前已确认进入阻尼模式，则继续保持阻尼模式。
- 若释放前未确认阻尼切换成功，则输出警告，要求人工检查。

### 4.5 调试模式识别

- 节点内置运控模式巡检线程。
- 可通过 `MotionSwitchClient::CheckMode` 判断机器人是否处于用户调试模式。
- 若机器人进入用户调试模式，高层急停链路被视为不可信。
- 若用户调试模式持续超过 5 秒，程序会播报“调试模式，急停无效”提示。
- 当急停触发但机器人仍处于调试模式时，程序会停止额外重试，并提示人工接管。

### 4.6 告警与人机提示

- 已集成 TTS 播报能力。
- 已集成 LED 灯光提示能力。
- 成功/失败通知颜色可通过参数配置。
- 对同类型通知做了冷却节流，避免频繁刷灯。
- 对启动自检场景采用“短超时 + 重试”策略，提高首次可感知提示成功率。

### 4.7 故障锁存与受控退出

- 支持 `SIGINT`、`SIGTERM` 信号受控处理。
- 当 GPIO 检测线程异常、检测器启动失败、自检失败时，会进入“保护失效”流程。
- 保护失效时会锁存故障、尝试切阻尼、播报警告、点亮失败灯、然后主动关闭 ROS2。
- 主循环支持节点异常后的重建和有限次重启。
- 若短时间内连续多次异常，则进程退出并返回失败码。

## 5. 软件架构说明

V1.0.0 采用“检测层 + 动作层 + 提示层 + 守护层”的结构：

![图1 系统架构图（分层结构）](<image/图1:系统架构图（分层结构）.png>)

*图 1. G1 急停程序的分层架构示意，展示检测、动作、提示与守护之间的职责划分。*

### 5.1 检测层

- 由 `EStopDetector` 负责。
- 处理 GPIO 初始化、输入线读取、边沿监听、回调分发、运行时错误上报。

### 5.2 动作层

- 由 `G1EmergencyStopNode::HandleEmergencyStopAction()` 负责。
- 统一串行处理急停事件，避免多个来源并发触发 `Damp`、TTS、LED。

### 5.3 提示层

- 由 `AudioClient` 提供 TTS 和 LED 接口。
- 用于启动成功、调试模式、急停失败、保护失效、异常退出等场景提示。

### 5.4 守护层

- `DetectorSupervisorLoop()` 负责 GPIO 检测器生命周期管理和健康监测。
- `MotionModeMonitorLoop()` 负责运控模式巡检。
- `main()` 中的外层循环负责节点异常重建与故障退出控制。

## 6. 关键参数说明

当前版本已支持以下关键参数：

| 参数名 | 说明 | 默认值 |
| --- | --- | --- |
| `debounce_ms` | 急停按钮消抖时间 | `20` |
| `active_low` | 是否低电平触发 | `false` |
| `max_retries` | 急停动作失败后的最大重试次数 | `2` |
| `retry_backoff_ms` | 重试退避时间 | `100` |
| `notification_cooldown_ms` | 同类灯光提示冷却时间 | `2000` |
| `success_led_r/g/b` | 成功提示灯颜色 | `0/255/0` |
| `failure_led_r/g/b` | 失败提示灯颜色 | `255/0/0` |

说明：当前急停成功后仍保持失败色红灯，这是刻意采用的现场保守策略，用于提示“急停仍处于按下期间，不可误判为已恢复正常”。

## 8. 已知边界与风险

### 8.1 已知边界

- 高层急停链路依赖机器人处于内置运控模式。
- 若机器人进入用户调试模式，高层服务可能不可用，此时只能提示人工接管。
- 当前版本未自动恢复运动，释放急停后仍需要人工确认和后续处置。
- 当前仓库缺少针对急停节点的单元测试与仿真测试。

### 8.2 风险说明

- `libgpiod` 缺失时，`g1_emergency_stop_node` 不会参与构建。
- 若现场 GPIO 接线与 `gpio_estop.hpp` 中的映射不一致，程序可能无法正确检测急停状态。
- 若高层服务链路不可用，程序虽会告警，但无法完全替代硬件级断使能保护。
